> 文档状态：初稿

> 更新日期：2026-07-10  
> 产品代号：Forkly

## 1. 文档目的

本文定义 Forkly 的产品目标、用户、核心流程、功能需求、权限模型、数据边界、技术约束、非功能要求和验收标准。

本文首先回答“产品为什么存在、为谁解决什么问题”，再约束“系统必须提供什么能力”。具体数据库字段、API 参数、页面视觉稿和开发任务拆分应在后续产品设计与技术设计文档中展开。

## 2. 产品概述

### 2.1 一句话定义

Forkly 是一套基于标准 Git、支持企业自托管的项目文件版本协作工具，让不会 Git 的成员也能在隔离工作区中修改代码、Markdown 和项目资料，并由负责人可视化审查后合并到受保护主线。

### 2.2 产品形态

Forkly 包含两个可独立部署或安装的产品：

#### 本地客户端

- Go 编写的常驻进程。
- 启动后驻留 Windows 系统托盘或 macOS 菜单栏。
- 不提供传统原生主窗口。
- 点击“打开控制台”后，使用系统默认浏览器打开本地 Web 控制台。
- 负责本地文件访问、Git 操作、凭据保存、文件监听、同步和文件关联。

#### 企业服务端

- Go 编写的自托管服务。
- 负责账号、组织、项目、权限、Git 远端、合并申请、审计、大文件和管理。
- 首期以单机 Docker Compose 部署为主。
- 通过 Web 管理界面和受保护 API 提供企业协作能力。

### 2.3 产品定位

Forkly 的核心不是重新实现 Git，也不是复制 GitLab 的全部功能，而是：

- 用普通用户能理解的语言包装 Git。
- 让负责人掌控唯一可信主线。
- 为多人和 AI 同时修改项目提供隔离、审查和责任链。
- 让企业能把代码、文档和常见项目资料保存在自己的基础设施中。

## 3. 背景与问题

AI 提高了内容、代码和方案的生成速度，也放大了企业内部原有的协作问题：

- 同一项目由多人和多个 AI 工具快速修改，版本来源变得混乱。
- 文件通过聊天、网盘和本地副本反复传递，缺少唯一可信版本。
- 负责人不知道修改了什么、为什么修改、谁负责、谁批准。
- Git 能解决历史和分支问题，但对非研发人员过于复杂。
- 传统网盘擅长同步，却不擅长结构化审查、主线保护和逐次变更追溯。
- GitHub、GitLab 等平台偏重研发流程，企业可能还存在私有部署、混合文件和简化体验需求。

Forkly 希望建立如下秩序：

```text
本地项目文件夹
    ↓ 保存版本
成员隔离工作区
    ↓ 提交负责人
可视化审查与审批
    ↓ 受控合并
唯一可信 main 主线
```

## 4. 产品目标与非目标

### 4.1 `1.0` 产品目标

1. 非 Git 用户无需命令行即可管理本地文件夹的版本。
2. 企业可在自己的服务器保存项目和协作元数据。
3. 多名成员能够在互相隔离的工作区中修改同一项目。
4. 普通成员无法直接破坏 `main`。
5. 负责人能够查看差异、审查并合并修改。
6. 代码、Markdown、图片、PDF 和常见二进制项目资料可被统一纳管。
7. 用户可把 Forkly 设为 Markdown 默认查看器和编辑器。
8. 管理员可以备份、恢复、升级、审计和诊断系统。
9. 底层项目始终保持标准 Git 兼容性，用户不会被专有格式锁定。

### 4.2 `1.0` 非目标

- 不自研 Git 替代协议、提交格式或仓库存储格式。
- 不提供完整 CI/CD、制品仓库、镜像仓库和云开发环境。
- 不提供 Google Docs/Notion 式实时多人共同编辑。
- 不提供 Office、PSD、视频等二进制文件的内容级自动合并。
- 不建设公共开源社区、代码市场或社交功能。
- 不以替代专业 IDE 为目标。
- 不在首个稳定版承诺多节点高可用和跨地域容灾。
- 不默认把企业文件发送给外部 AI 服务。

### 4.3 `1.1` 之后的方向

- AI 修改来源与责任人追踪。
- AI 辅助提交说明、合并摘要和风险提示。
- 任务、需求、审批模板和项目看板。
- OIDC、SAML、LDAP、SIEM 与企业消息系统集成。
- 跨项目资产、策略和 Agent 治理。

## 5. 目标用户

### 5.1 项目负责人

典型特征：

- 对项目最终内容和主线质量负责。
- 可能了解 Git，也可能只理解“正式版”和“草稿”。
- 需要决定成员权限、审查修改、处理争议和回溯责任。

核心诉求：

- 确保未经审查的修改不能进入主线。
- 快速理解一次修改涉及哪些文件和内容。
- 知道谁发起、谁审查、谁合并，以及是否有 AI 参与。

### 5.2 项目成员

典型特征：

- 使用代码编辑器、Office 软件或普通文件工具修改项目。
- 不一定理解分支、暂存、rebase 等 Git 概念。

核心诉求：

- 像使用普通文件夹一样工作。
- 不担心自己的修改覆盖他人内容。
- 能清楚看到“未保存版本、未上传、待同步、存在冲突”等状态。
- 用简单步骤把修改交给负责人。

### 5.3 审查者或维护者

典型特征：

- 帮助负责人审查专业内容或维护项目。

核心诉求：

- 聚合查看待审查修改。
- 评论、要求修改、批准或合并。
- 确认新提交后旧批准是否仍有效。

### 5.4 只读成员

典型特征：

- 需要查看、下载或跟踪项目，但不应修改。

核心诉求：

- 安全获取可信主线。
- 查看历史和文档，不误触写操作。

### 5.5 企业系统管理员

典型特征：

- 负责部署、账号、安全、备份、升级和故障处理。

核心诉求：

- 数据存储在企业控制范围内。
- 权限可回收、操作可审计、故障可恢复。
- 客户端和服务端版本兼容关系明确。

## 6. 产品原则

### 6.1 用户目标优先于 Git 术语

默认界面使用“项目、我的工作区、保存版本、提交负责人、同步主线”。原始分支名、提交 SHA 和 Git 错误在高级信息中保留。

### 6.2 本地优先，但不是自动网盘同步

- 用户在真实本地文件夹中工作。
- 本地修改先由用户明确保存为 Git 版本，再上传工作区。
- 默认不把每次文件保存自动提交或自动合并。
- 可以自动检测变化和远端状态，但高影响写操作必须让用户确认。

### 6.3 主线可信

`main` 代表负责人认可的正式状态。普通成员只能通过合并申请进入主线。

### 6.4 失败必须可理解、可恢复

系统不能只显示原始 Git 报错。必须说明：

- 当前发生了什么。
- 用户内容是否安全。
- 下一步可以做什么。
- 如何中止或获取诊断信息。

### 6.5 标准与可迁移

- 内容与历史保存在标准 Git 仓库中。
- 大文件使用 Git LFS 兼容方式。
- 即使停用 Forkly，企业仍可使用标准 Git 工具读取仓库。

### 6.6 人对主线负责

AI 可以总结、建议和执行受限任务，但不能绕过权限、审查和主线保护。

## 7. 核心概念

### 7.1 项目

一个受 Forkly 管理的本地文件夹及其对应 Git 仓库。`1.0` 中一个项目对应一个 Git 仓库。

### 7.2 主线

项目默认分支 `main`。代表当前正式、可信、可供所有成员同步的版本。

### 7.3 我的工作区

普通用户看到的个人修改空间。底层是从 `main` 创建的独立 Git 分支。

内部建议按用户和设备隔离：

```text
users/{userId}/{deviceId}/{workspaceId}
```

这样同一用户在多台电脑上工作时不会让一个长期分支互相覆盖。内部名称默认不暴露给普通用户。

### 7.4 保存版本

把选中的本地文件修改记录为 Git commit。保存版本不等于已经上传，也不等于已经进入主线。

### 7.5 上传工作区

把本地工作分支的提交推送到服务端对应分支。

### 7.6 合并申请

成员请求把工作区修改合入 `main` 的审查对象。它包含差异、提交、评论、批准和合并记录。

### 7.7 同步主线

获取最新 `main`，并安全地更新当前工作区。实现可以使用 merge；普通用户不需要理解内部命令。

### 7.8 项目负责人

对项目成员、配置、主线和最终合并负责的角色。项目任何时刻必须至少有一名负责人。

## 8. 核心状态模型

产品必须明确区分以下状态，不能统一显示成含糊的“已同步”：

### 8.1 文件状态

- 无本地修改
- 有未保存版本的修改
- 有冲突
- 文件被其他成员锁定

### 8.2 本地版本状态

- 本地无新提交
- 有尚未上传的本地提交
- 本地工作区与远端工作区一致

### 8.3 主线关系

- 工作区基于最新主线
- 主线有新版本待同步
- 工作区和主线已分叉
- 当前修改可自动合并
- 当前修改存在冲突

### 8.4 审查状态

- 未提交负责人
- 草稿
- 等待审查
- 要求修改
- 已批准
- 已合并
- 已拒绝
- 已关闭

界面必须始终告诉用户：内容目前只在本地、已经上传到个人工作区，还是已经进入正式主线。

## 9. 端到端核心流程

## 9.1 首次启动客户端

1. 用户安装并启动 Forkly。
2. 客户端建立单实例锁并启动本地 Go 服务。
3. 托盘出现图标。
4. 用户点击“打开控制台”。
5. 客户端生成短期会话，使用系统浏览器打开本地地址。
6. Web 控制台验证会话后显示项目首页。
7. 首次使用时检查 Git 运行环境并给出明确结果。

成功结果：用户进入本地控制台，客户端继续在后台运行。

## 9.2 把本地文件夹纳入版本管理

1. 用户选择“新建本地项目”或“添加现有文件夹”。
2. 新建时选择父目录并输入项目名；添加时通过系统文件夹选择器选择现有目录。
3. 客户端规范化路径并检查访问权限、符号链接和 Git 状态。
4. 若目录不是 Git 项目，用户确认后初始化。
5. 系统建议生成 `.gitignore`，但不擅自忽略未知文件。
6. 用户查看变化，填写说明并保存第一个版本。

成功结果：项目文件仍保留在原目录，同时拥有可查看的 Git 历史。

## 9.3 发布到企业服务器

1. 用户配置企业服务器地址并登录。
2. 用户选择已有本地项目，点击“发布到服务器”。
3. 选择组织、项目名称和可见范围；创建人自动成为首位项目负责人。
4. 服务端创建项目、bare repository 和受保护 `main`。
5. 客户端建立远端并上传现有历史。
6. 服务端校验默认分支和权限。

成功结果：项目可以从其他授权设备恢复，主线与本地历史一致。

## 9.4 邀请成员并拉取项目

1. 负责人邀请成员并分配角色。
2. 成员登录客户端，在“可加入项目”中看到项目。
3. 成员选择本地目录并拉取。
4. 服务端从最新 `main` 为该用户和设备创建工作分支。
5. 客户端检出该分支，并在界面显示“我的工作区”。

成功结果：成员获得独立可写空间，但不能直接修改服务端 `main`。

## 9.5 修改、保存和上传

1. 成员使用任意本地软件修改文件。
2. 客户端检测文件变化并刷新状态。
3. 成员查看文本差异或二进制元数据。
4. 成员选择文件，填写版本说明并保存版本。
5. 客户端提示“已在本地保存，尚未上传”。
6. 成员点击“上传工作区”。

成功结果：修改已安全上传个人工作区，不影响主线。

## 9.6 发起与处理合并申请

1. 成员选择“提交给负责人”。
2. 系统检查是否有未上传提交、是否落后主线、是否存在冲突。
3. 成员填写标题、说明和审查人。
4. 负责人查看文件、差异、提交和评论。
5. 负责人批准、要求修改或拒绝。
6. 满足规则后，由服务端执行合并。
7. 合并申请记录最终主线提交。
8. 成员客户端收到结果，引导同步并创建下一工作区。

成功结果：主线只包含经过授权流程的修改，且责任链完整。

## 9.7 同步与冲突恢复

1. 客户端检测主线领先。
2. 用户点击“同步主线”。
3. 系统先检查未提交修改并创建必要恢复点。
4. 无冲突时完成更新。
5. 有冲突时进入冲突向导，禁止继续上传或发起可合并状态。
6. 用户解决、继续或安全中止。

成功结果：无论成功、中止还是失败，用户原始内容都有明确状态和恢复路径。

## 9.8 双击 Markdown 文件

1. 用户把 Forkly 设置为 `.md` 默认应用。
2. 用户双击 Markdown 文件。
3. 操作系统把文件打开事件发送给客户端；若客户端已运行，则转发给现有实例。
4. 客户端规范化并授权路径，生成短期 `fileId`。
5. 客户端打开本地 Web 控制台的 Markdown 页面。
6. 页面通过 `fileId` 请求内容并安全渲染。
7. 若文件属于已管理项目，同时显示项目、工作区、修改和历史状态。

成功结果：URL 不暴露完整路径，页面不能借此读取其他任意文件。

## 10. 功能需求

需求优先级定义：

- P0：`1.0` 上线必需。
- P1：重要增强，可在相应路线图版本进入。
- P2：远期方向，不阻塞 `1.0`。

## 10.1 本地客户端与托盘

### FR-CLIENT-001 单实例运行（P0）

- 同一用户会话只允许一个 Forkly 主进程。
- 第二次启动、协议链接或文件打开请求必须转交给现有实例。
- 进程异常退出后，锁必须可被安全回收。

### FR-CLIENT-002 托盘能力（P0）

托盘菜单至少包含：

- 打开控制台
- 当前账号和服务器
- 最近项目
- 同步状态摘要
- 暂停或恢复后台检查
- 查看日志
- 检查更新
- 退出

### FR-CLIENT-003 本地 Web 控制台（P0）

- 仅绑定回环地址，不能默认监听局域网接口。
- 前端静态资源随客户端发布，不依赖公网 CDN。
- 浏览器会话必须有短期令牌，并校验来源和请求防伪信息。
- 关闭浏览器不退出后台客户端。
- 退出客户端后，本地 API 必须不可访问。

### FR-CLIENT-004 自动打开与端口管理（P0）

- 客户端自行选择可用端口，不能因默认端口被占用而无法启动。
- 不把可长期复用的高权限令牌直接放在 URL 查询参数中。
- 浏览器启动失败时提供可复制的本地地址和处理说明。

### FR-CLIENT-005 文件变化检测（P0）

- 监听已打开项目变化，并以防抖方式刷新 Git 状态。
- 文件监听事件丢失时必须通过周期性或用户触发的完整刷新纠正。
- 大量变化不能无限并发启动 Git 进程。

### FR-CLIENT-006 本地设置（P0）

- 保存项目列表、界面设置、服务器信息和非敏感偏好。
- 访问令牌必须进入 Windows Credential Manager、macOS Keychain 或等价安全存储。
- 配置损坏时可以备份旧文件并恢复最小可用设置。

### FR-CLIENT-007 Git 运行时（P0）

- 普通用户安装 Forkly 后即可使用，不要求提前手工安装或配置 Git。
- Windows 安装程序内置经过验证的 Git 运行时；若采用受控下载，则必须校验来源、签名或哈希。
- macOS 发布方案必须提供同等的一步可用体验，不能把开发者工具安装错误直接留给普通用户处理。
- 高级用户可选择兼容的系统 Git，系统应显示实际使用版本。
- Forkly 未经用户明确确认不得修改系统级或全局 Git 配置。
- 分发 Git 运行时必须遵守其开源许可证和源代码提供义务。

## 10.2 本地项目与 Git

### FR-GIT-001 添加和初始化项目（P0）

- 支持在选定父目录下新建项目文件夹，也支持选择普通文件夹或已有 Git 仓库。
- 新建时校验名称、目标路径和重名，不覆盖已有目录内容。
- 初始化前必须说明将创建 `.git` 目录。
- 不能删除、移动或批量修改用户现有文件。
- 支持中文、空格、Unicode 和平台允许的长路径。

### FR-GIT-002 状态查看（P0）

- 显示新增、修改、删除、重命名、未跟踪、忽略和冲突状态。
- 支持按目录和状态筛选。
- 超大目录或特殊文件错误不能让整个项目页面失效。

### FR-GIT-003 差异查看（P0）

- 文本文件支持逐行差异。
- 图片支持前后预览。
- 二进制文件只显示版本元数据，不尝试文本渲染。
- 大差异必须分页或按文件延迟加载，并设置安全上限。

### FR-GIT-004 保存版本（P0）

- 用户可选择一个或多个文件保存版本。
- 必须填写或确认版本说明。
- 保存前显示将被包含的文件。
- 保存成功后显示提交标识和同步状态。
- 首期不要求行块级暂存，但架构不得阻止后续增加。

### FR-GIT-005 历史查看（P0）

- 显示提交时间、作者、说明和提交标识。
- 查看某次提交修改的文件和内容。
- 按文件查看历史。
- 支持复制原始 Git SHA。

### FR-GIT-006 Git 操作执行安全（P0）

- Go 端使用明确参数调用 Git，不经过 Shell 拼接。
- 每个仓库的写操作串行化。
- 所有长操作支持超时、进度或等待状态。
- 应用关闭时优雅停止；无法立即停止的操作需标记恢复状态。
- 原始 stdout/stderr 只作为诊断信息，不直接作为普通用户文案。

### FR-GIT-007 高风险操作（P0）

- 丢弃修改、删除分支、回退版本等必须二次确认。
- 默认优先创建可恢复记录，而不是永久删除。
- 强制推送、hard reset 和历史改写不向普通成员开放。

### FR-GIT-008 项目文件浏览（P0）

- 展示当前工作区的目录树，并支持按名称查找文件。
- 文本、图片、PDF 和 Markdown 使用相应安全预览；未知或超大文件只显示元数据。
- 从页面打开系统文件管理器或外部编辑器时必须使用本地受信进程执行。
- 文件树不得越过项目根目录，也不得通过符号链接读取未授权路径。

## 10.3 服务端部署与远程仓库

### FR-SERVER-001 自托管部署（P0）

- 提供 Linux x86-64 的容器部署方式。
- 最小依赖包含 Forkly Server、PostgreSQL 和持久化 Git 数据卷。
- 引入 LFS 后增加 S3 兼容对象存储。
- 配置通过环境变量或受保护配置文件注入。
- 提供健康检查、启动检查和版本信息接口。

### FR-SERVER-002 管理员初始化（P0）

- 首次启动必须通过一次性流程创建首位系统管理员。
- 初始化完成后自动关闭公开初始化入口。
- 忘记密码或管理员恢复使用离线管理命令，不提供未授权后门。

### FR-SERVER-003 项目远端（P0）

- 每个项目对应一个隔离的 bare Git repository。
- 支持 Git Smart HTTP clone、fetch 和 push。
- 每次访问都进行账号、项目、分支和操作权限校验。
- 不能仅依靠仓库目录权限保护主线。

### FR-SERVER-004 项目生命周期（P0）

- 创建、查看、编辑基本信息、归档、恢复和软删除。
- 创建项目的用户自动成为首位项目负责人。
- 项目任何时候都不能处于零负责人的有效状态；负责人离开前必须先转移职责。
- 删除前展示影响范围并要求高权限确认。
- 项目不能在存在未完成备份或迁移时被静默清除。

### FR-SERVER-005 数据一致性（P0）

- Git 仓库是文件内容与提交历史的事实来源。
- PostgreSQL 是账号、权限、项目元数据、审查和审计的事实来源。
- 数据库不得维护一份易与 Git 分叉的完整文件内容镜像。
- 合并等跨 Git 与数据库操作必须具有幂等键、补偿和一致性检查。

## 10.4 账号、组织与权限

### FR-IAM-001 账号认证（P0）

- 支持管理员创建账号或邀请注册。
- 支持密码登录、会话退出、令牌吊销和登录限速。
- `1.0` 支持可选多因素认证。
- `1.3` 再增加企业单点登录。

### FR-IAM-002 组织（P0）

- 项目必须属于一个组织。
- 组织管理员管理组织成员和默认策略。
- 组织成员身份不自动等于所有项目可见。

### FR-IAM-003 项目角色（P0）

角色及核心权限如下：

| 操作 | 负责人 | 维护者 | 成员 | 只读成员 |
| --- | --- | --- | --- | --- |
| 查看与拉取项目 | 是 | 是 | 是 | 是 |
| 修改自己的工作区 | 是 | 是 | 是 | 否 |
| 发起合并申请 | 是 | 是 | 是 | 否 |
| 评论合并申请 | 是 | 是 | 是 | 可配置 |
| 批准合并申请 | 是 | 是 | 否 | 否 |
| 合并到 `main` | 是 | 可配置 | 否 | 否 |
| 管理普通成员 | 是 | 可配置 | 否 | 否 |
| 修改主线规则 | 是 | 否 | 否 | 否 |
| 归档或删除项目 | 是 | 否 | 否 | 否 |

- 所有权限由服务端执行。
- 项目必须至少保留一名负责人。
- 负责人离开前必须转移所有权。

### FR-IAM-004 最小权限令牌（P0）

- 浏览器会话、客户端访问令牌、Git 凭据和 API Token 分开管理。
- 令牌可按设备查看和吊销。
- 令牌不得写入项目仓库、日志或浏览器持久化明文存储。

## 10.5 工作区与同步

### FR-WORKSPACE-001 自动工作区（P0）

- 成员首次在设备上拉取项目时自动创建工作区。
- 工作区必须从当时最新 `main` 创建。
- UI 默认显示“我的工作区”，高级信息可查看内部 Git 分支名。
- 服务端限制成员只能写入自己的分支命名空间。

### FR-WORKSPACE-002 多设备隔离（P0）

- 工作区至少由用户、设备和工作区标识共同唯一确定。
- 同一用户在多台设备上的未完成修改互不覆盖。
- 用户可在账号安全页面识别和撤销设备。

### FR-WORKSPACE-003 工作区轮换（P0）

- 合并完成后，客户端在工作目录干净时从最新 `main` 创建下一工作区。
- 若仍有未合并或未提交修改，不得自动删除或重置原工作区。
- 旧服务端分支根据保留策略归档或删除，合并申请历史必须保留。

### FR-SYNC-001 获取远端状态（P0）

- 显示本地与远端工作区、工作区与 `main` 的领先/落后数量。
- 后台只做轻量状态检查，不自动改变用户文件。
- 网络离线时显示最后成功检查时间。

### FR-SYNC-002 上传与拉取（P0）

- 上传前确认目标是当前用户工作区。
- 非快进推送默认拒绝，禁止普通用户强制推送。
- 拉取或同步前检查未提交修改。
- 网络中断后可安全重试。

### FR-SYNC-003 同步主线（P0）

- 无冲突时一键把最新 `main` 更新到工作区。
- 有冲突时进入显式处理状态。
- 不允许通过隐式 reset 丢弃工作区提交。

## 10.6 合并申请与主线保护

### FR-MR-001 创建合并申请（P0）

- 来源只能是用户有权使用的工作区。
- 目标默认且首期固定为 `main`。
- 包含标题、说明、提交列表、文件差异、发起人和时间。
- 无实际差异时不得创建。

### FR-MR-002 差异审查（P0）

- 按文件延迟加载差异。
- 支持文本行级查看、图片对比和二进制元数据。
- 明确标识重命名、大文件、生成文件和无法预览文件。
- 显示工作区相对当前 `main` 是否已过期。

### FR-MR-003 评论与审查（P0）

- 支持总体评论和文件/行级评论。
- 审查结果包括批准、要求修改和拒绝。
- 工作区新增提交后，旧批准按项目规则自动失效。
- 评论删除或编辑必须留下审计信息。

### FR-MR-004 合并检查（P0）

合并前至少检查：

- 操作者权限
- 合并申请状态
- 所需批准数量
- 来源分支未被替换
- 目标 `main` 未发生未评估变化
- 是否可自动合并
- 文件锁和服务端策略

### FR-MR-005 合并策略（P0）

- `1.0` 默认使用 squash merge，使主线保持清晰。
- squash 提交信息包含合并申请编号和原作者信息。
- 服务端是最终合并执行者。
- 合并接口必须幂等，重复请求不能重复生成主线提交。
- 合并成功后记录源分支、源提交、目标提交、发起人、批准人和合并人。

### FR-MR-006 主线保护（P0）

- 普通成员不得通过 Git HTTP 或 API 直接更新 `main`。
- 删除、重建和强制更新主线默认禁止。
- 负责人也应通过合并申请进入主线；紧急旁路若未来提供，必须高风险确认并完整审计。

## 10.7 冲突与恢复

### FR-CONFLICT-001 冲突检测（P0）

- 上传、同步和合并前检测分叉和冲突风险。
- 服务端无法自动合并时不得进入 `main`。
- UI 显示冲突文件与类型。

### FR-CONFLICT-002 冲突处理（P0）

- 文本文件提供基础三方内容查看。
- 用户可选择当前工作区版本、主线版本或外部编辑器。
- 处理过程中持续检测文件状态。
- 用户可继续或中止，且两条路径都有明确结果。

### FR-CONFLICT-003 异常恢复（P0）

- 启动时检测 merge、rebase、cherry-pick 等未完成状态，即使这些状态由外部工具产生。
- 不自动猜测并执行破坏性修复。
- 提供恢复建议、原始诊断和项目备份入口。

## 10.8 混合文件、LFS 与锁

### FR-FILE-001 文件分类（P0）

- 按内容和扩展名判断文本、图片、PDF、二进制和未知文件。
- 不能仅因扩展名未知就尝试把巨大文件加载到内存。
- 设置单文件预览和差异大小上限。

### FR-LFS-001 Git LFS（P0）

- 服务端实现兼容的鉴权、上传、下载和对象校验。
- 客户端检测 Git LFS 环境，并在需要时自动配置或提供安装引导。
- 失败不能留下指向不存在对象的可合并提交。

### FR-LFS-002 对象存储（P0）

- 支持 S3 兼容对象存储。
- 数据库只保存必要元数据，不保存大文件正文。
- 通过内容哈希校验对象完整性。
- 支持分片、重试、容量限制和垃圾回收。

### FR-LOCK-001 二进制文件锁（P1）

- 用户可对不可合并文件申请锁。
- 其他成员看到锁持有人和时间。
- 超时、离职和管理员强制解锁必须有策略与审计。
- 锁是降低冲突的协作提示，不替代服务端最终合并检查。

## 10.9 Markdown 查看与编辑

### FR-MD-001 文件关联（P0）

- 支持在 Windows 和 macOS 注册 Forkly 为 `.md` 打开方式。
- 用户可以选择设为默认应用，不能在未同意时强制抢占默认关联。
- 卸载时正确移除自身关联，不破坏其他应用设置。

### FR-MD-002 安全打开文件（P0）

- 接收操作系统文件打开事件。
- 规范化真实路径并处理符号链接。
- 项目内文件按项目权限打开。
- 项目外文件仅进入明确的临时本地模式，不自动加入项目或上传。
- 浏览器只接收短期 `fileId`，不能传任意路径读取文件。

### FR-MD-003 Markdown 渲染（P0）

- 支持 CommonMark 常见语法、任务列表、表格、代码块和目录。
- 支持项目内相对链接和相对图片。
- 默认禁用原始 HTML，或经过严格白名单清洗。
- 禁止脚本、事件属性、危险 URL Scheme 和任意本地文件跳转。
- 外部图片和链接采用清晰策略，企业可默认禁用外部资源。

### FR-MD-004 Markdown 编辑（P0）

- 支持源码、分屏和预览模式。
- 自动保存只写入本地文件，不自动生成 Git 提交。
- 清楚展示“文件已保存”和“版本已保存”的差异。
- 外部程序修改同一文件时，禁止静默覆盖。
- 支持从编辑页查看差异、保存版本和发起合并申请。

### FR-MD-005 项目附件（P1）

- 粘贴或拖入图片时保存到可配置项目附件目录。
- 自动使用相对路径。
- 文件名冲突时不得覆盖已有附件。

## 10.10 管理、审计与通知

### FR-ADMIN-001 管理控制台（P0）

- 查看和管理组织、用户、项目、配额、会话和系统状态。
- 高风险管理操作需要重新确认身份。
- 系统管理员不因角色身份自动获得所有项目内容权限；需要明确授权或受审计的支持流程。

### FR-AUDIT-001 审计事件（P0）

至少记录：

- 登录、退出、失败登录、令牌创建和吊销
- 用户、角色和权限变化
- 项目创建、归档、恢复和删除
- Git 推送拒绝与主线更新
- 合并申请状态、批准和合并
- 文件锁、管理员解锁
- 备份、恢复、升级和关键配置变化

每条事件至少包含：

- 事件类型
- 时间
- 操作者及其身份类型
- 组织和项目范围
- 目标对象
- 请求或关联操作标识
- 结果
- 必要且脱敏的前后状态

### FR-NOTIFY-001 通知（P0）

- 站内通知覆盖邀请、要求审查、要求修改、批准、合并和失败。
- 支持邮件作为首个外部通知渠道。
- 用户可以设置非安全类通知偏好。
- 权限回收、安全告警等关键通知不能被普通用户完全关闭。

### FR-BACKUP-001 备份与恢复（P0）

- 备份集覆盖 PostgreSQL、Git 仓库、LFS 对象和关键配置。
- 提供一致性检查，发现缺失 Git 或 LFS 对象。
- 支持自动计划、保留策略、加密和异地保存说明。
- 恢复流程必须可在测试环境演练。

### FR-UPDATE-001 更新（P0）

- 客户端检查签名更新并提示安装。
- 服务端升级前执行兼容与备份检查。
- 数据迁移可重复检测，失败时停止进入不兼容版本。
- 明确客户端与服务端支持的版本范围。

## 10.11 AI 能力（`1.1`，P1）

### FR-AI-001 来源声明

- 版本或合并申请可声明人工、AI 辅助、Agent 或脚本来源。
- 记录发起责任人、工具、模型标识、任务和时间。
- 不默认保存提示词、模型密钥和完整私有上下文。

### FR-AI-002 辅助审查

- AI 可生成版本说明、合并摘要、影响范围和审查清单。
- AI 输出必须明确标识并由人确认。
- AI 不具备绕过主线策略的权限。
- 外部模型不可用时，核心 Git 与审查功能正常工作。

### FR-AI-003 Agent 身份

- 自动化 Agent 使用独立服务身份和最小权限令牌。
- 每次写操作关联具体任务和授权人。
- 企业可以要求 Agent 变更经过额外人工批准。

## 12. 技术架构约束

本文不是详细技术设计，但以下约束直接影响产品安全和可行性。

## 12.1 本地客户端建议结构

```text
SystemTray
    ↓
GoClientProcess
    ├── LocalHTTPServer
    ├── EmbeddedWebAssets
    ├── GitOperationQueue
    ├── FileWatcher
    ├── CredentialAdapter
    ├── FileAssociationAdapter
    └── ServerAPIClient
            ↓
        LocalProjects
```

- Web 前端建议采用 React、TypeScript 和 Vite，构建后嵌入 Go 发布物。
- 托盘、凭据、文件关联和通知通过平台适配层隔离。
- 不使用 PowerShell、Bash 等 Shell 字符串拼接作为核心业务执行方式。
- 所有文件系统权限由本地 Go 进程持有，浏览器页面只能使用受限 API。

## 12.2 服务端建议结构

```text
ReverseProxyTLS
    ↓
ForklyGoServer
    ├── WebAndRESTAPI
    ├── AuthAndRBAC
    ├── GitSmartHTTP
    ├── MergeService
    ├── AuditService
    └── BackgroundJobs
        ├── PostgreSQL
        ├── BareGitRepositories
        └── S3CompatibleObjectStorage
```

- `1.0` 优先模块化单体，不提前拆分微服务。
- Git 读取和写入任务必须有并发上限。
- 仓库路径由内部 ID 映射，不直接信任用户输入路径。
- 主线保护必须同时覆盖 Web API 和 Git 传输入口。
- 异步任务需要持久状态、重试上限和幂等标识。

## 12.3 数据事实来源

- Git：文件内容、树、提交、分支和标签。
- PostgreSQL：用户、组织、权限、设备、项目元数据、工作区映射、合并申请、评论、审查、通知和审计。
- 对象存储：LFS 大文件正文。
- 本地配置：项目路径、UI 偏好、服务器列表和缓存。
- 操作系统安全存储：客户端敏感令牌。

数据库实体只引用 Git commit、tree、branch 等标识，不复制全部仓库历史。

## 12.5 接口边界

- 本地受限接口建议使用 `/local-api/v1/`。
- 企业业务接口建议使用 `/api/v1/`。
- Git 传输使用独立明确的 Smart HTTP 路径。
- 长操作进度可使用 SSE；确有双向实时需求时再引入 WebSocket。
- 所有写接口支持请求标识；合并、上传确认等关键接口必须幂等。

## 13. 安全与隐私要求

## 13.1 本地控制台安全

- 仅监听 `127.0.0.1` 或 `::1`，默认不暴露局域网。
- 校验 Origin、Host、CSRF 信息和本地会话。
- 使用短期、不可预测、最小权限的浏览器会话。
- 文件接口使用服务端生成的能力标识，不接受任意绝对路径。
- 路径必须规范化并解析符号链接，拒绝逃逸授权项目根目录。
- 对 Markdown、SVG、HTML 和外部 URL 做严格内容安全控制。

## 13.2 服务端安全

- 生产部署必须使用 TLS。
- 密码使用适合密码存储的强哈希算法。
- 登录、令牌和高成本接口限速。
- 服务端密钥不得进入日志或仓库。
- Git 命令参数化执行，禁止 Shell 注入。
- 仓库、LFS 和 API 使用同一权限事实来源。
- 主线保护使用服务端钩子或等价强制机制，不能只依靠 UI。
- 发布物需校验来源和签名。

## 13.3 隐私

- 默认不收集项目文件内容、文件名清单或提交正文到产品方。
- 自托管版遥测默认关闭或明确征得管理员同意。
- 诊断包默认脱敏令牌、密码、远端 URL 凭据和用户目录。
- AI 功能必须在发送前展示数据范围和目标服务。
- 管理员配置数据保留期时，不得破坏法定或项目既定审计要求。

## 14. 非功能需求

## 14.1 可靠性

- 任何失败不得静默丢弃用户文件、未提交修改或已有 Git 提交。
- Git 写操作必须按仓库串行化。
- 服务端关键写操作必须可重试或具有明确补偿。
- 应用异常退出后能检测未完成状态。
- 备份恢复后必须验证 Git 引用、数据库关联和 LFS 对象完整性。

## 14.2 性能初始目标

以下是需要通过基准环境校准的产品目标，不构成未经测试的 SLA：

- 客户端冷启动到托盘可用：参考设备上目标不超过 3 秒。
- 打开本地控制台到首屏可交互：目标不超过 2 秒。
- 普通项目状态刷新：目标 1 秒内出现反馈，长操作持续显示进度。
- 支持至少 20,000 个受跟踪文件和 20,000 次提交的常见项目，不让 UI 主线程长时间无响应。
- 合并差异按文件延迟加载，不一次性把超大 patch 送入浏览器。
- Git、预览和异步任务均设置并发上限，防止请求风暴拖垮设备或服务器。

## 14.3 可扩展性

- `1.0` 聚焦单节点部署。
- Git 仓库、数据库和对象存储路径彼此独立，允许后续扩容。
- 后台任务不可只存于进程内存。
- 项目 ID 与物理路径解耦，为后续迁移和分片留出空间。

## 14.4 兼容性

- `0.1` 首发 Windows 11。
- `1.0` 正式支持 Windows 11 和仍处于安全支持期的 macOS 版本。
- 服务端正式支持主流 Linux x86-64 容器环境。
- Git 最低和推荐版本在发布时依据测试矩阵确定。
- 路径、大小写、文件权限、换行符和文件锁定差异必须进入跨平台测试。

## 14.5 可用性

- 核心流程不要求用户理解 Git 命令。
- 高风险操作使用结果导向的中文说明。
- 每个长任务提供等待、取消或后台运行状态。
- 关键操作支持键盘导航、清晰焦点和屏幕阅读语义。
- 错误信息包含用户可执行建议和可展开诊断详情。

## 14.6 可观测性

- 服务端输出结构化日志、健康检查和基础指标。
- 每次关键请求和后台任务有可关联标识。
- 客户端保留有限周期滚动日志。
- 日志不得记录密码、访问令牌和未脱敏项目内容。
- 管理员可查看队列积压、失败任务、存储容量和备份状态。

## 14.7 可维护性

- 客户端与服务端采用明确版本协商。
- 数据库迁移必须可测试、可重复识别并在失败时停止。
- 平台相关能力使用适配层，不把 Windows 或 macOS 实现散落在业务代码中。
- API、Git 规则和权限判断必须有自动化测试。

## 15. 数据、备份与删除策略

### 15.1 默认数据保留

- 主线和已合并历史默认永久保留，除非企业明确执行合规删除流程。
- 合并申请、评论、批准和审计记录按管理员策略保留。
- 未合并工作分支在提醒后按策略归档，再进入可删除状态。
- LFS 对象只有在确认无任何可达提交引用且超过保留期后才能清理。

### 15.2 软删除

- 项目删除首先进入软删除状态。
- 软删除期间禁止普通访问，但负责人和管理员可查看恢复期限。
- 永久删除必须确认备份策略、引用关系和审计要求。

### 15.3 备份一致性

完整备份必须包含：

1. PostgreSQL。
2. bare Git repositories。
3. LFS 对象存储。
4. 服务端密钥以外的必要配置与版本信息。

密钥应通过企业密钥管理独立备份，不应明文混入普通数据包。

## 16. 边界情况

系统设计和验收至少覆盖以下情况：

- 项目路径包含中文、空格、Emoji 或较长层级。
- 用户把项目目录移动、重命名或删除。
- 用户同时使用 Forkly 和其他 Git 客户端。
- 外部工具切换分支或留下未完成 merge/rebase 状态。
- `.git` 目录损坏、磁盘空间不足或文件被占用。
- 服务端在 push、LFS 上传或合并中途重启。
- 数据库记录存在但 Git 仓库缺失，或相反。
- Markdown 文件引用项目外路径、符号链接或恶意 SVG。
- 同一用户多设备同时修改同一文件。
- 成员在合并申请进行中被移出项目。
- 审查通过后来源工作区新增提交。
- 负责人离职但项目仍需保留负责人。
- 大文件上传完成但提交未推送，或提交已到达但 LFS 对象缺失。
- 客户端版本高于或低于服务端支持范围。
- 系统时间不准确；审计以服务端时间为准。

## 17. 成功指标

## 17.1 激活指标

- 安装后成功打开本地控制台的用户比例。
- 成功添加第一个项目的用户比例。
- 新用户完成第一次“保存版本”的中位时间。
- 首次发布或拉取远程项目的成功率。

## 17.2 核心价值指标

- 活跃项目每周保存版本次数。
- 成员成功发起合并申请的比例。
- 负责人完成审查和合并的比例。
- 从发起到完成合并的中位耗时。
- 通过 Forkly 进入 `main` 的变更占主线变更比例。

## 17.3 质量护栏

- 已确认数据丢失事件数：目标为 0。
- 未授权主线更新事件数：目标为 0。
- Git 操作失败率和恢复成功率。
- 客户端崩溃会话比例。
- 备份恢复演练成功率。
- 审计关键事件完整率。

## 17.4 易用性指标

- 因分支、暂存、推送等 Git 术语产生的求助数量。
- 用户能否正确区分“文件已保存、版本已保存、已上传、已进入主线”。
- 冲突流程的安全中止率与解决率。
- Markdown 双击打开成功率和打开耗时。

## 18. `1.0` 发布验收

只有同时满足以下条件，才能标记为 `1.0`：

### 产品闭环

- 本地文件夹可以初始化、查看变化、保存版本和查看历史。
- 项目可以发布到服务端并在另一设备拉取。
- 负责人可以邀请成员并分配角色。
- 成员自动进入隔离工作区，无法直接推送 `main`。
- 成员可上传、发起合并申请，负责人可审查和合并。
- 合并后其他成员可同步到相同主线。
- Markdown 可以双击打开、渲染和编辑，并进入版本流程。

### 数据与恢复

- 没有未解决的高风险数据丢失缺陷。
- 客户端异常退出、网络中断和服务端重启均有验证过的恢复结果。
- 数据库、Git 和 LFS 的完整备份恢复至少演练一次。
- 升级迁移在真实规模测试数据上验证。

### 权限与安全

- Git HTTP、业务 API 和 Web UI 的权限结果一致。
- 主线保护无法被普通成员旁路。
- 本地 API 不能被普通恶意网页直接调用。
- Markdown 不能执行任意脚本或越权读取本地文件。
- 敏感令牌不出现在仓库、URL、普通日志和前端持久化明文中。

### 运维与支持

- 有安装、升级、备份、恢复、监控和故障排查文档。
- 客户端和服务端兼容范围明确。
- Windows 与 macOS 发布物完成签名或公证流程。
- 管理员能导出脱敏诊断信息和审计记录。

## 19. 主要风险与应对

### 19.1 产品范围变成“另一个 GitLab”

风险：不断加入 issue、CI、制品、Wiki 等功能，核心体验迟迟不可用。

应对：

- `0.4` 前只围绕本地版本、成员隔离和主线合并。
- 非核心能力优先集成现有系统。
- 每个版本必须对应一个用户问题和退出标准。

### 19.2 长期用户分支积累冲突

风险：每个用户只有一条永久分支，多个任务和设备混在一起。

应对：

- 用户、设备、工作区三级隔离。
- 合并后轮换工作区。
- 后续任务功能允许一项任务对应一个工作区。

### 19.3 混合文件拖垮 Git

风险：视频、压缩包和 Office 文件快速扩大仓库，且无法自动合并。

应对：

- 尽早识别二进制和大文件。
- `0.6` 引入 LFS、对象存储、容量限制和文件锁。
- 不承诺二进制内容级差异与合并。

### 19.4 本地 Web 控制台扩大攻击面

风险：恶意网页请求本地端口，或利用文件参数读取任意文件。

应对：

- 只监听回环地址。
- 短期会话、来源校验、CSRF 防护。
- 文件能力 ID、规范路径和项目根目录授权。
- Markdown 严格清洗和内容安全策略。

### 19.5 Git 命令和跨平台差异

风险：Windows/macOS 路径、锁、换行和 Git 版本导致行为不一致。

应对：

- 参数化执行，不依赖 Shell 语法。
- 建立跨平台端到端测试仓库。
- 明确支持的 Git 版本并提供诊断。

### 19.6 Git 与数据库状态不一致

风险：合并时服务中断，Git 已更新但合并申请状态未更新。

应对：

- 幂等操作、预期提交校验和恢复任务。
- Git 作为内容事实来源，数据库保存引用。
- 定期一致性检查和人工修复工具。

### 19.7 AI 功能带来隐私与错误信任

风险：企业内容被意外发送外部服务，或负责人过度相信 AI 摘要。

应对：

- AI 在 `1.1` 后单独引入，默认关闭。
- 明确数据发送范围和保留策略。
- AI 输出始终标记，并需要人确认。
- AI 不拥有绕过主线审批的权限。

## 20. 待用户验证的产品假设

以下内容已有默认方案，但必须通过原型访谈和试点验证：

1. 非研发用户是否能理解“主线”和“我的工作区”。
2. 用户是否接受“文件保存”和“保存版本”是两个动作。
3. 负责人更偏好 squash merge，还是希望保留成员的每次小提交。
4. 项目中大文件类型、平均大小和修改频率。
5. Markdown 是否能成为需求、方案、会议纪要等企业文档的推荐格式。
6. 客户最需要的是纯内网部署、私有云部署，还是两者都要。
7. Windows 与 macOS 用户比例及首发优先级。
8. 企业更重视任务流程，还是 AI 修改来源追踪。

这些假设不应通过堆叠功能验证，而应使用可操作原型、真实文件夹和完整合并任务进行观察。

## 21. 建议的 MVP 验证任务

在 `0.4` 完成后，邀请目标用户完成以下任务，不提供 Git 命令提示：

1. 把一个已有文件夹加入 Forkly 并保存第一个版本。
2. 把项目发布到测试服务器。
3. 邀请另一名成员。
4. 成员拉取、修改两个文件、保存并上传。
5. 成员把修改提交给负责人。
6. 负责人发现其中一个不合适的修改并要求调整。
7. 成员修正后再次提交。
8. 负责人批准并合并。
9. 第三名成员同步并确认获得最新主线。
10. 制造一次文本冲突，观察用户是否理解状态并能安全中止。

验证重点不是用户是否“会 Git”，而是用户是否始终知道：

- 自己改了什么。
- 内容现在位于哪里。
- 谁能看到。
- 是否已经进入主线。
- 失败后内容是否安全。
