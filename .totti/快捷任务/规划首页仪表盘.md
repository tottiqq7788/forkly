基于 **当前已实现能力** 与 **近期可落地的新能力**，重新规划首页「数据概览」仪表盘：只保留对用户真正有决策价值的指标，允许删改现有不重要统计项，禁止堆砌图表。

## 产品边界（规划时必须遵守）

当前 0.1 是 **本机多项目可视化 Git**，首页应回答三件事：

1. 现在有多少工作要处理？（待保存、阻断）
2. 最近有没有在持续保存版本？（活跃度）
3. 整体健康吗？（目录丢失、无法读取、已暂停保存）

**不要**为以下能力预留空壳指标（尚未实现或 0.1 明确不做）：

- 待上传 / 待同步 / Push·Pull
- 冲突审查队列、合并申请、组织成员贡献
- 账号 / 服务器在线状态
- 长期代码行数趋势、扩展名占比、作者排行、雷达图、桑基图、分支网络图

### 2. 评估当前仪表盘每一项（可删可改）

对 **现有** 首页模块逐项判定：`保留` / `改文案或降级` / `删除` / `合并进其他模块`。

当前实现（规划时以代码为准，可能已变化）大致包括：

- KPI：项目总数、版本总数、近 30 天新增版本、待保存文件
- 图：近 30 天版本活动柱状图
- 图：项目状态分布分段条
- 图：待保存变更类型水平条

评估标准：

- **行动价值**：用户看完会不会去「项目」页处理某件事？
- **准确性**：是否会被分页上限、静默失败、空仓误导？
- **成本**：是否需要扫全仓库历史、N+1 请求？
- **重复**：是否与另一指标说同一件事？
- **认知负担**：非 Git 用户能否一眼看懂？

明确建议示例方向（供执行时判断，不是必须照抄）：

- 「版本总数」可能信息密度低 → 可改为「近 7/30 天」为主，总数降级或去掉
- 「变更类型」细分（copied / typechange）可能过细 → 可只保留 modified / untracked / deleted / conflicted 等
- 「项目状态分布」若与 KPI 高度重复 → 可合并或改成「需要关注」列表（仍可不展示项目路径以外的操作入口，按用户既有约束）

### 3. 选择性提出「值得新增」的指标（宁缺毋滥）

最多提出 **0–3 个** 新增项。每个新增必须写清：

- 用户问题（一句话）
- 数据来源（现有 API / 需小改后端 / 需大改）
- 展示形式（数字卡 / 条 / 柱 / 短列表）
- 为什么比现有某项更重要
- 若不做会有什么损失

禁止为了「看起来像仪表盘」而凑饼图、雷达图、多系列折线。

### 4. 给出目标信息架构（一版即可）

用简洁结构描述目标首页，例如：

```text
标题：数据概览
├─ KPI 行（≤4）
├─ 主图 / 主模块（1 个最重要）
├─ 次要模块（0–2 个）
└─ 空状态 / 部分失败说明
```

并注明：

- 首页是否继续 **不展示** 项目列表、路径、添加项目入口（默认：是，与现产品决策一致；若建议改回，必须单独论证）
- 窄屏（约 792px 主区）与深色主题如何排布
- 哪些点击应下钻到 `/projects/:id` 的变更或历史 Tab（若保留下钻）

### 5. 落地成本分级

把方案拆成：

- **P0**：只改前端文案/删模块/重排（不改 API）
- **P1**：小改 `ProjectView` 或 `dashboard/activity` 字段
- **P2**：新聚合逻辑或新接口

默认推荐优先可在 P0/P1 完成的方案。
